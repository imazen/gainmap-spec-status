# Cross-codec gain map test vectors

Small files only (<30 KB each). All have provenance in
[`manifest.toml`](manifest.toml) with source, license, and SHA-256.

## Layout

```
test-vectors/
├── jpeg/        UltraHDR JPEGs (hdrgm XMP + MPF + appended gain map JPEG)
├── avif/        AVIF with `tmap` derived image item (HEIF Amd 1)
├── jxl/         Synthetic `jhgm` box payloads (libjxl golden layout)
├── heic/        (TODO — no samples yet, see gaps below)
├── png/         (N/A — spec not merged)
├── tiff/        (N/A — no spec binding)
└── sources/     Standalone ISO 21496-1 metadata blobs (parser fixtures)
```

## What we have

| Category | Files | Size range | Source |
|---|---|---|---|
| JPEG UltraHDR | 5 | 149 B – 24 KB | `~/work/codec-corpus/ultrahdr-conformance/` (awesome-gain-maps, libultrahdr-testdata, edge-cases, invalid) |
| AVIF `tmap` | 5 | 1.3 KB – 3.1 KB | `~/work/libavif/tests/data/` (BSD-2) |
| JXL `jhgm` | 2 | 154 B – 173 B | generated by `tools/gen-jhgm-golden.py` matching libjxl's `gain_map_test.cc` golden |
| ISO 21496-1 | 2 | 61 B – 62 B | generated by `tools/gen-iso21496.py`, both JpegApp2 and AvifTmap wire variants |

## What we're missing (gaps)

1. **HEIC Apple aux-type gain map** — need a small iPhone HEIC file
   carrying `urn:com:apple:photo:2020:aux:hdrgainmap`. The smallest
   iPhone HEIC files are ~800 KB. Options:
   - Build one with a libheif tool from an SDR source + synthetic gain map
   - Extract a crop from a real iPhone HEIC
2. **HEIC with HEIF Amd 1 `tmap`** — need a libheif-produced (or Apple iOS
   18+) file. Not yet available.
3. **Real libjxl-produced `jhgm` box** — we only have synthetic fixtures
   matching the libjxl test layout. Need a file produced by `cjxl --ultrahdr`
   against a real UltraHDR JPEG for byte-exact conformance.
4. **PNG** — no test vector exists because the W3C `gMAP` proposal has
   not merged. Will add one when the 4th edition draft publishes a
   reference file.
5. **Apple JPEG gain map (new and old)** — `libavif/tests/data/apple_gainmap_new.jpg`
   and `.../apple_gainmap_old.jpg` are 50 KB each (over our 30 KB rule).
   Need user confirmation to include, or crop them first.

## Regeneration

Synthetic vectors are reproducible:

```bash
# JXL jhgm golden (matches libjxl gain_map_test.cc)
python3 tools/gen-jhgm-golden.py test-vectors/jxl/jhgm_golden.bin

# ISO 21496-1 parser fixtures (JpegApp2 and AvifTmap)
python3 tools/gen-iso21496.py test-vectors/sources/
```

SHA-256s in `manifest.toml` should match the generated files exactly.
If they drift, either the generator changed or the dependency did — bisect.

## How zen crates consume these

Tests in each zen crate should load from `../gainmap-spec-status/test-vectors/`
(sibling repo). Suggested pattern:

```rust
// zencodec, zenavif, zenjxl, zenjpeg — parser-level
let bytes = include_bytes!("../../gainmap-spec-status/test-vectors/sources/iso21496_jpeg.bin");
let params = zencodec::gainmap::parse_iso21496_fmt(bytes, Iso21496Format::JpegApp2).unwrap();
assert_eq!(params.channels.len(), 1);
assert_eq!(params.channels[0].linear_max(), 2.0f64.powf(1.3));
```

```rust
// zenavif-parse — full-file
let avif = std::fs::read("../gainmap-spec-status/test-vectors/avif/seine_sdr_gainmap_srgb.avif").unwrap();
let parsed = zenavif_parse::parse(&avif).unwrap();
assert!(parsed.gain_map.is_some());
```

Do NOT duplicate the files into each crate's test tree. Load via relative
path or symlink.

## Do not bloat this

Every file added must answer one of:

- exercises a branch no other file exercises
- is from an upstream reference implementation
- is synthetic and reproducible from `tools/`

Otherwise the corpus drifts into a generic image collection.
